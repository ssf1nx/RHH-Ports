[Dusklight v1.2.0](https://github.com/TwilitRealm/dusklight/releases/tag/v1.2.0)'s headline on the release notes is the one many will ask about: **experimental OpenGL ES and D3D11 backends**. The [previous post](https://github.com/JeodC/RHH-Ports/blob/main/docs/devlog/2026-05-12-dusklight.md) called out Aurora being Vulkan-only as the reason most handhelds in the catalog can't run Dusklight, so the obvious question is whether v1.2.0 unlocks anything. Short answer: it doesn't.

The actual change in the Aurora submodule ([encounter/aurora@1000661](https://github.com/encounter/aurora/commit/1000661ee493f248b8597e4dfa1d2871d76a1d9)) flips `DAWN_ENABLE_OPENGLES` on only inside an `if (ANDROID)` branch. The Linux branch still hardcodes it off. `BACKEND_OPENGLES` is now referenced in Aurora's runtime backend list, but no Linux build can actually emit a GLES adapter today.

Two more things stack against forcing it on in this repo's build:

**Build cost.** The prebuilt `dawn-linux-aarch64.tar.gz` is compiled with GLES disabled. Turning it on would mean switching the Dusklight Docker recipe from the prebuilt Dawn package to a vendor build (Dawn-from-source via FetchContent), which adds ~30–60 minutes to every Dusklight CI build and pulls in Dawn's full dependency fetch (abseil, tint, glslang, SPIRV-Tools, the lot). This wouldn't matter to me usually, but there's more.

**Aurora's surface code.** [BackendBinding.cpp](https://github.com/encounter/aurora/blob/10006618/lib/dawn/BackendBinding.cpp) only knows how to bind a wgpu surface to an X11 or Wayland SDL window — `kmsdrm` falls through to `return nullptr` regardless of backend. The Vulkan build has the same constraint; it's just invisible because every device that ships a Vulkan-capable driver also ships X11, Wayland, Sway, or xWayland. A handheld stuck on kmsdrm could in theory wrap the port in a Westonpack-style nested Wayland compositor, but that's a compositor-overhead penalty stacked on top of the GLES-translation penalty, on a device class that's already CPU-bound.

The performance ceiling is the part where this all gets clearest anyway. Dawn's GLES path translates WGSL to GLSL ES via Tint and recompiles shaders per-pipeline on the driver side. That's measurably slower than Vulkan on the same gpu. The Adreno driver issues that TwilitRealm is still fighting on the Android side aren't OS-specific; same Snapdragon behaves the same on Linux. RK3588 + Mali-G610 already runs Vulkan; everything weaker than it can't keep up with the reimplementation on CPU regardless of which graphics API the renderer uses. There's no device class that lights up only when GLES becomes available.

What v1.2.0 *does* bring that's worth the bump: the Goron Mines elevator crash fix, autosave safety (no autosave after loading a state or resetting), a real framerate cap option, and an "Area" output resampler that's smoother than bilinear when downscaling from the internal resolution.
